# AGENTS.md

This file applies to AI coding agents such as Codex and GitHub Copilot across projects. Use it for general workflow, safety, validation, and communication standards. Combine it with any repository-specific, path-specific, or tool-specific instruction files that also apply.

## Instruction Precedence

Apply instructions in this order when they conflict:

1. Direct user task
2. Deeper path-specific `AGENTS.md`
3. Matching project instruction files, such as `instructions/*.instructions.md`
4. This shared `AGENTS.md`
5. General tool or agent defaults

When instructions overlap, prefer the most specific rule that applies to the files being changed.

If two instructions at the same scope conflict and the correct interpretation is not obvious, stop and ask for clarification instead of guessing.

## Mission and Scope

- Make the smallest safe change that solves the user request.
- Prefer root-cause fixes over quick patches.
- Keep changes focused.
- Do not refactor unrelated code.
- Do not commit, push, create branches, publish releases, or deploy unless explicitly asked.
- Preserve existing behavior unless the requested change requires otherwise.

## Repository Awareness

- Read relevant files before editing.
- Understand the existing project structure before making changes.
- Prefer editing source, tests, documentation, configuration, and scripts.
- Avoid changing generated, vendored, build output, cache, or transient files unless the task explicitly targets them.
- Do not edit files marked as generated, such as files containing `// <auto-generated>`, unless the task explicitly requires regenerating or modifying generated output.
- Treat vendored or external dependency folders as read-only unless the task explicitly requires changes there.
- Follow existing naming, architecture, formatting, and project conventions.

## Execution Workflow

1. Clarify the requested outcome from the prompt and existing code.
2. Inspect relevant files before making changes.
3. Confirm assumptions from code, tests, configuration, or documentation.
4. Implement the smallest targeted change.
5. Validate with the narrowest relevant checks first.
6. Run broader checks when the scope or risk justifies it.
7. Summarize what changed, how it was validated, and any remaining risks.

## Code Change Guidelines

- Match existing style, naming, and architecture.
- Avoid introducing new dependencies unless necessary.
- Preserve public APIs unless a breaking change is explicitly requested.
- Keep scripts and automation idempotent where practical.
- Avoid one-letter variable names except for conventional short-lived loop or lambda variables.
- Do not add inline comments unless they clarify non-obvious behavior or the user requested them.
- Prefer readable, maintainable code over clever shortcuts.
- Keep error handling structured and consistent with the project.
- Use logging consistently with existing project patterns.
- Do not log secrets, credentials, tokens, passwords, or sensitive user-provided values.
- Validate external input at trust boundaries.
- Keep changes easy to review.

## Testing and Validation Standards

- Use repository-native commands and scripts rather than ad-hoc alternatives when available.
- Run targeted tests, lint, formatting, type checks, or builds appropriate to the change.
- Start with the narrowest relevant validation and expand when needed.
- Add or update automated tests when behavior changes, bugs are fixed, or user-visible logic is introduced.
- If tests are not practical, document the validation gap and why coverage was not added.
- Report validation failures clearly.
- Do not fix unrelated failures unless asked.
- Documentation-only or instruction-only changes usually do not require a build or test run.

## Security and Privacy

- Never print, hardcode, commit, or expose secrets, tokens, client secrets, SAS values, passwords, certificates, private keys, or connection strings.
- Use configuration files, secret stores, and environment variables consistently with existing project patterns.
- Preserve and improve secure defaults.
- Sanitize and validate user inputs.
- Avoid logging raw user input when it may contain sensitive data.
- Treat audit logs as sensitive data.
- Avoid weakening authentication, authorization, encryption, cookie security, CSRF protection, CORS restrictions, or input validation.
- Do not make destructive infrastructure, data, or cloud operations without explicit user intent.
- Follow least-privilege principles.
- Prefer secure-by-default implementations.

## Dependency and Tooling Policy

- Prefer stable, supported versions of packages, SDKs, tools, drivers, modules, and CLIs.
- Before introducing or upgrading dependencies, check compatibility with the current project runtime and deployment constraints.
- Prefer security and patch upgrades promptly when compatible.
- Avoid pinning to outdated versions without a documented reason.
- Update lock files, manifests, and related configuration consistently.
- Validate dependency changes with appropriate tests or builds.
- Do not add a dependency for functionality that can be implemented simply and safely with existing project tools.

## Documentation and Change Tracking

Update documentation when behavior, setup, configuration, deployment, authentication, operations, or user-facing workflows change.

Common documentation targets include:

- `README.md`
- `docs/`
- `CHANGELOG.md`
- `RELEASE_NOTES.md`
- Operational runbooks
- Configuration examples
- Developer setup instructions

Keep documentation:

- Accurate
- Concise
- Task-oriented
- Aligned with the actual command surface and repository layout

If there is no documentation impact, state that in the handoff summary.

## Deployment and Operations Hygiene

- Use existing project scripts for provisioning, build, release, deployment, and local execution when available.
- Keep deployment changes idempotent and parameterized.
- Avoid destructive operations unless explicitly requested.
- Document operational changes that affect setup, deployment, rollback, monitoring, or maintenance.
- Validate infrastructure or deployment changes with the safest available checks before applying them.
- Do not assume cloud provider, environment, subscription, tenant, region, or account details. Confirm from existing configuration or the user request.

## Release Hygiene

When asked to prepare or verify a release:

- Keep version metadata aligned across code, release notes, changelog, manifests, and tags.
- Ensure release artifacts are reproducible from source.
- Store generated release artifacts in the repository's established artifact location when one exists.
- Verify release tags, release objects, artifact attachments, and download links when applicable.
- Do not mark release work complete until requested release verification is complete or clearly documented as pending.

## Analyzer, Lint, and Formatting Policy

- Prefer fixing analyzer, lint, formatting, and type-check findings over suppressing them.
- Keep suppressions narrow and documented when they are necessary.
- Use project-level configuration for shared rules.
- Use path-scoped configuration for localized exceptions.
- Do not reformat unrelated files unless the task explicitly requires it.

## Communication and Handoff

Be concise, explicit, and action-oriented.

When handing off work, include:

- What changed
- What was validated
- What was not validated, if anything
- Known risks or assumptions
- Next logical steps when useful

Do not overstate certainty. Clearly distinguish verified facts from assumptions.

## Default Engineering Preferences

- Prefer PowerShell examples for cross-platform automation unless the repository clearly uses another shell or toolchain standard.
- Prefer reusable functions for scripts.
- Prefer structured error handling.
- Prefer clear logging over silent failure.
- Prefer repository-native conventions over personal preferences.
- Prefer minimal targeted edits over broad rewrites.
